# CAMP Product Guide

This guide describes CAMP from a product and user workflow perspective. It covers the daily use of Data Analytics, report governance, assignments, self-service reports, record review, Team Management, and ML Workflow Studio.

## 1. Platform Overview

CAMP is organized into three primary workspaces.

| Workspace | Purpose |
|---|---|
| Data Analytics | SQL analysis, report definition, assignments, dispatcher monitoring, self-service reporting, transaction analysis, record review, and dashboards. |
| Team Management | Task portfolio management, team ownership, progress tracking, task history, dashboards, role permissions, and expense tracking. |
| ML Workflow Studio | Governed model workflow design with MDS inputs, Python processing steps, artifacts, metrics, UAT/Prod lifecycle, and MDL outputs. |

All workspaces share the same platform concepts: authenticated users, role-based permissions, service-account execution, centralized metadata, audit logs, and operational traceability.

## 2. Users, Roles, and Access

Users sign in through the configured enterprise login flow. The first approved administrative user manages user status, roles, permissions, source visibility, and platform settings.

Typical role groups are:

| Role | Product position |
|---|---|
| Administrator | Manages platform configuration, users, roles, sources, reports, assignments, scheduler, and operational settings. |
| Analyst | Designs analytical reports, manages report metadata, reviews executions, and supports assignment operations. |
| User | Runs allowed self-service reports, reviews assigned records, and works on assigned tasks. |
| Pending | Created user identity waiting for approval. |

Permissions are enforced both in the interface and service layer. Screen visibility alone is not treated as the security boundary.

## 3. Data Sources

Data source profiles represent controlled analytical sources such as SQL Server and Oracle databases. The central CAMP database is a platform component and is not added as a normal analytical source.

A source profile defines connection type, server context, database or service identity, allowed use, and access scope. Source access can be restricted by role, user, server, database, schema, or object-level rules depending on the deployment policy.

## 4. SQL Editor

The SQL Editor is used to prepare analytical logic against authorized sources. It supports single-source work and managed multi-source analytical flows.

In single-source mode, the query runs directly against the selected SQL Server or Oracle source. This mode is suitable for native SQL logic, temporary tables, common table expressions, stored procedures allowed by policy, and source-local analysis.

In multi-source mode, the user explicitly controls which source each analytical step uses and which narrow result should be moved into the central working area. This provides governed multi-source analysis while keeping data movement intentional and traceable.

SQL Editor output can be used for direct inspection, report definition, self-service report logic, MDS dataset preparation for ML workflows, and operational validation.

## 5. Report Management

Report Management turns analytical logic into governed report assets.

A report definition contains:

- report code and report family,
- report name, category, owner, and description,
- source mode and selected sources,
- SQL text or managed source flow,
- parameters for self-service scenarios,
- reference mappings,
- methodology and operational notes,
- activation state and version history.

### Report Families

| Family | Purpose |
|---|---|
| FR | Recurring fraud or exception scenario output. |
| SR | Recurring process, control, or monitoring scenario output. |
| SSR | Self-service report requested by users through parameters. |
| MDS | Model dataset source used by ML Workflow Studio. |
| MDL | Model output package generated by ML Workflow Studio. |

FR and SR reports are assignable recurring outputs. SSR reports are requested on demand. MDS reports are not assigned to users; they feed model workflows. MDL reports are assignable model outputs and may flow into Record Review.

### Versioning and Activation

Report definitions are versioned. When SQL, source selection, parameters, references, or key metadata changes, a new version is created. Executions remain linked to the report version active at the time of execution.

Reports can be active or inactive. Inactive reports remain historically visible but are not expected to be used for new recurring assignments unless reactivated according to governance rules.

## 6. Assignments

Assignments connect report outputs to operational ownership.

An assignment defines:

- report to be executed or distributed,
- primary responsible user,
- CC users,
- period and schedule rule,
- notification behavior,
- export format,
- execution or distribution note,
- whether result rows should be transferred to Record Review.

FR and SR reports can have independent assignment periods. Each assignment can define its own responsible user, notification, export, and review settings.

MDL outputs produced by the same ML workflow can be assigned to different users. However, the model execution period is shared across the model group. Changing the period for one active MDL output in the same model group updates the shared period rule for the active outputs in that group.

The assignment-level review option determines whether execution rows are transferred to the Record Review queue. If review routing is disabled, the result remains available as a report result without creating row-level review work items.

## 7. Dispatcher and Scheduled Execution

The dispatcher evaluates due assignments and executes workloads through the controlled service layer.

For standard reports, due assignments are executed according to their own schedule. For ML-origin MDL outputs, due assignments are grouped by workflow/model so the workflow runs once, produces the required MDL packages, and applies assignment-specific distribution and review behavior.

Dispatcher history allows users to inspect run status, start/end time, row counts, errors, retry status, and execution context.

## 8. Self-Service Reporting

Self-service reports allow authorized users to request predefined parameterized reports without modifying report logic.

A typical self-service flow:

1. User selects an available SSR report.
2. User enters required parameters and request reason.
3. Request is queued for controlled execution.
4. The service layer runs the report.
5. Output is delivered or made available according to platform configuration.
6. Request and execution history remain visible for audit and support.

SSR reports are not recurring assignments. They are user-initiated controlled requests.

## 9. Transaction Analysis

Transaction Analysis provides search and drill-down over persistent analytical result sets. Users can search by customer, user, accounting reference, unit, report, date, or other configured dimensions.

The purpose is to support investigation and cross-report analysis without requiring the user to reconstruct SQL logic manually.

## 10. Record Review

Record Review is the operational workbench for assigned report or model result rows.

Review users can:

- open assigned result sets by period and report,
- inspect row-level details,
- enter risk classifications and explanations,
- save drafts,
- perform bulk updates where allowed,
- finalize reviewed rows,
- view review history.

Finalized reviews are locked according to governance rules. Review history preserves the relationship between report execution, result row, assigned user, review action, and completion state.

## 11. Dashboard

Dashboards provide operational visibility across analytical and review processes. Typical dashboard areas include report execution status, assignment status, review completion progress, open and closed review counts, task summaries, overdue items, and model output indicators.

Dashboard content is intended for operational monitoring rather than replacing detailed module screens.

## 12. System Administration Screens

Administrative screens include central database status, data source management, user approval, roles, source access rules, SMTP settings, MFA configuration, scheduler settings, logs, and maintenance actions made available by policy.

Administrative functions should be used by authorized operators only.

## 13. Team Management

Team Management is designed for structured work portfolios such as audit, control, investigation, or operational follow-up tasks.

| Concept | Description |
|---|---|
| Work year | Annual planning and reporting context. |
| Plan period | Annual plan, quarterly plan, or in-year additional task context. |
| Task | A defined assignment with scope, status, dates, owner, and team. |
| Team lead | Person responsible for coordination and progress updates. |
| Team member | User assigned to work on the task. |
| Department | Organizational unit associated with the task. |
| Progress | Central percentage and working summary indicating task advancement. |
| History | Status, ownership, scope, and decision trail. |

A task typically moves from assigned to in progress, then to approval, suspension, completion, closure, or archive depending on workflow rules. Status transitions are recorded so the task portfolio can be reviewed historically.

Working summaries allow team members or team leads to record progress, blockers, scope notes, and completion explanations. Extensions and suspensions are recorded in task history. The dashboard summarizes open tasks, completed tasks, progress levels, status distribution, priority distribution, team workload, and year-based portfolio indicators.

Expense tracking supports task-related travel or daily allowance tracking where applicable. It is integrated with user profile and task context so expenses remain associated with the relevant work item.

## 14. ML Workflow Studio

ML Workflow Studio is the model workflow governance area of CAMP. It connects MDS datasets, Python processing steps, generated artifacts, model metrics, and MDL publish outputs into a traceable canvas.

The module is not intended to replace a Python IDE. Model code can be developed in external tools. Studio governs how model workflows are connected to enterprise data inputs, run history, artifacts, model versions, and assignable analytical outputs.

### UAT and Prod

| Mode | Behavior |
|---|---|
| UAT | Used for experimentation, trial runs, validation, and temporary outputs. UAT runs can be cleaned. Publish validation does not create final operational MDL distribution. |
| Prod | Used for finalized model workflows. Runs, artifacts, metrics, model versions, published MDL reports, assignments, and review outputs are preserved as operational records. |

New model work starts in UAT. A successful UAT run can be promoted to Prod, creating a finalized model version and preserving the required snapshot.

### Canvas Design

The visual canvas is the workflow. Nodes and edges represent the actual model flow. Users add, move, delete, connect, and configure nodes directly on the canvas.

Common node stages include MDS Dataset, Data Preparation, EDA, Variable Analysis, Feature Engineering, Split, Training, Evaluation, Scoring, and Publish.

A model objective such as anomaly scoring, classification, regression, time series, segmentation, churn, fraud/risk scoring, recommendation, or blank canvas can prepare an initial diagram that the user can customize.

### MDS Inputs

Raw data is prepared in Data Analytics and saved as an MDS report. In Studio, an MDS Dataset node connects the model workflow to that governed dataset.

MDS is not assigned directly to business users. It is the model input. Its schedule is not independent from the model; when a Prod model is triggered by due MDL assignments, the MDS step runs as part of that same model run.

### Python Steps, Artifacts, and Metrics

Python nodes can use inline script content or a script file under the model workspace. Each model can have its own isolated runtime so model dependencies do not affect the CAMP application runtime.

Python steps may produce standard files such as metrics, model cards, artifacts, scored outputs, and publish contracts. CAMP catalogs recognized outputs automatically.

Artifacts are system-cataloged outputs generated by workflow runs. Users do not manually edit artifacts through the artifact list. Metrics are stored as structured values so model performance and run quality can be reviewed historically.

### MDL Publish Outputs

The Publish node converts model results into one or more MDL report packages. Each package can represent a different output segment, risk group, business queue, or review population.

One model run may produce multiple MDL packages for different units or review teams. These packages appear in Data Analytics Assignments and can be assigned to different responsible users while sharing the same model period.

### Scheduling and Distribution

ML Studio designs the workflow. Data Analytics Assignments manage schedule and ownership for Prod MDL outputs.

- Studio defines how the model produces results.
- Assignments define who receives which result and when it should run.
- Dispatcher runs the model once per due period for the workflow group.
- Publish writes the MDL outputs.
- Assignment settings determine notification, export, and review routing.

If an MDL assignment has review routing enabled, its result rows become Record Review items. If disabled, the MDL result remains available as a persistent report result without row-level review tasks.

## 15. Product Workflow Examples

### Recurring Control Report

1. Analyst prepares SQL logic.
2. Analyst saves it as an FR or SR report.
3. Administrator or authorized analyst creates an assignment.
4. Dispatcher runs the report according to schedule.
5. Results are delivered or routed to Record Review.
6. Responsible users complete review actions.
7. Dashboard and logs provide status visibility.

### Self-Service Report

1. Administrator defines an SSR report with parameters.
2. User opens Self-Service Reports.
3. User enters parameters and submits a request.
4. Service layer executes the report.
5. Output is delivered according to configured behavior.

### ML Model Output Review

1. Raw dataset logic is saved as MDS.
2. ML Studio workflow uses the MDS node and Python steps.
3. UAT runs validate the model.
4. Successful workflow is promoted to Prod.
5. Publish node creates one or more MDL outputs.
6. MDL outputs are assigned in Data Analytics Assignments.
7. Dispatcher triggers the model once when due.
8. MDL outputs are published and routed according to assignment settings.
9. Review users complete row-level review where enabled.

| Term | Meaning |

| SR | Process or control scenario report. |
| SSR | Self-service report. |
| MDS | Model dataset source used as an ML workflow input. |
| MDL | Model output report package produced by an ML workflow. |
